Introduction

The Plan is a simple, single player economics simulation game set in a world where your goal is to survive a global economic catastrophe and rebuild.

User Interface

The user interface of the system is based on a simple rule-engine based system that lays on top of react. Currently this is semi-experimental, but I may make further tweaks to use it in other games.

The Core Operating System

At the core of the actual experience is the user interface, which is defined by the rule engine framework above. In essence it creates a simple operating system of sorts that the player uses in the game. The operating system is not defined by a set of individual files, but rather assumes that everything in the operating system is a fact.

;;;; Message Client
;; This is the message client, which is sort of like a cross between email
;; and regular messages.

;; An address which may be used by a person.
(defrecord Address

     display-name])                     ; The display name of the user

;; A thread of messages that reply to each other.
(defrecord MessageThread
    [id                              ; unique ID of the message thread
     addresses                       ; list of addresses involved
     name])                          ; an optional name for the thread

;; A single message in the thread
(defrecord Message
    [id                    ; unique ID of the message
     thread                ; the ID of the thread it replies to
     reply-to              ; the ID of the message it replies to (optional)
     time                  ; the time the message was sent
     address               ; the ID of the address sending the message
     contents])            ; the contents of the message

(defrecord MessageDraft
    [id                                 ; The ID of this draft
     thread                             ; The thread it replies to
     reply-to                           ; The specific person it replies to
     contents])                         ; The contents of the message

(defrule message-disp
  [Message [{:keys [thread reply-to time address contents]}]]
  =>
  (insert!
   (com :div {:class     "message"
              :thread-id thread}
        (com :div {:class "message-header"}
             (com :div reply-to)
             (com :div time)
             (com :div address))
        (com :div {:class "message-contents"}
             contents))))

(defrule message-thread-disp
  "Get a set of messages in a single thread."
  [Thread (= id ?id)]
  [?draft <- MessageDraft [{:keys [thread reply-to contents]}] (= thread ?id)]
  [?messages <- (acc/all) :from [Component
                                 (= (:class attr) "message")
                                 (= (:thread-id attr) ?id)]]
  =>
  (insert!
   (com
    (apply com :div {:class "message-history"}
           ?messages)
    (com :input
         {:class     "reply-draft"
          :type      "text"
          :value     ""
          :on-change #(alter-extern! ?draft
                                     (->MessageDraft ?id thread reply-to %))})
    (com :input
         {:class    "reply-button"
          :type     "button"
          :value    "Reply"
          :on-click #(insert-extern! (->Message (random-uuid)
                                                (:id ?user-id)
                                                ()))}))))

Defining Our Economic Model





So to start out the game start of defining our model. For the model we use a sort of hybrid approach, wherein the model uses agents for certain high-level figures, and rough models for more complex systems such as the whole economy. This is to save computational resources as actually modeling millions of citizens doing things would be expensive (and likely inaccurate).





However, it should be noted that in this economy we do still have tens of thousands of discrete products, which is partially to make the game feel more alive (just looking into the economy should make you feel slightly dizzy) but also to make it so that you have to delegate many tasks, use markets, or use some complicated algorithmic planning scheme. 2 Or some byzantine mixture of the above, which is both what you start with and the most likely outcome given the nature of the game.





By default knowledge about the mechanisms of the economy is incomplete. While the economy does have a definite state in the game, the player will never truly know what the economy is doing at each point, and the actions the player takes can cause them to understand the economy more or less. Likewise, the player can lie about the state of the economy to others, though this may become more or less possible depending on the actions of the player.





During each round of the game the real state of the economy is computed. This is done in multiple stages, first with the moves of various major economic actors (the agents), followed by the economic simulation itself executing accordingly. Following this, the display is produced using the "fudge-factor" for various industries.
